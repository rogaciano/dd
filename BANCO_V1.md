# Banco V1

## Objetivo

Este documento descreve a proposta inicial de banco para a V1, com base em [MODELO_DOMINIO.md](e:\projetos\disque\MODELO_DOMINIO.md), [V1_ESCOPO.md](e:\projetos\disque\V1_ESCOPO.md) e [STACK.md](e:\projetos\disque\STACK.md).

O foco e sair do legado para um modelo:

- simples de manter
- consistente com `Laravel + PostgreSQL`
- seguro por padrao
- preparado para crescer sem herdar os acoplamentos do sistema antigo

---

## Diretrizes Gerais

### Banco

- `PostgreSQL 16`
- schema inicial: `public`

### Convencoes

- nomes de dominio em portugues
- nomes sem acentos
- tabelas em `snake_case`
- FKs com sufixo `_id`
- datas em `timestamptz`
- textos longos em `text`
- dados variaveis em `jsonb`

### Excecoes praticas

Para reduzir atrito com Laravel, vale manter:

- `id`
- `created_at`
- `updated_at`
- termos tecnicos como `checksum` e `mime_type`

### Chaves primarias

Padrao recomendado:

- `bigint generated by default as identity`

### Identificadores publicos

Nao expor `id` interno para o usuario final.

Usar:

- `protocolo` para acompanhamento
- `token_acompanhamento_hash` para validacao segura

### Estados e tipos controlados

Para a V1, prefira:

- `varchar` + validacao na aplicacao
- `check constraints` nos campos criticos

Evitar:

- `enum` nativo do PostgreSQL cedo demais

## Tabelas da V1

## Bloco 1: Acesso e Governanca

### `usuarios`

Finalidade:

- usuarios internos do backoffice

Colunas principais:

- `id bigint pk`
- `nome varchar(150) not null`
- `login varchar(80) not null unique`
- `email varchar(150) null unique`
- `senha varchar(255) not null`
- `status varchar(20) not null`
- `ultimo_login_em timestamptz null`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Checks sugeridos:

- `status in ('ativo', 'inativo', 'bloqueado')`

Indices:

- `unique(login)`
- `unique(email)` com cuidado se `email` for opcional
- `index(status)`

### `papeis`

Finalidade:

- papeis de acesso

Colunas principais:

- `id bigint pk`
- `nome varchar(80) not null`
- `slug varchar(80) not null unique`
- `descricao text null`
- `ativo boolean not null default true`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Exemplos:

- `administrador`
- `supervisor`
- `analista`
- `atendente`
- `visualizador`

### `papel_usuario`

Finalidade:

- associacao N:N entre usuario e papel

Colunas principais:

- `usuario_id bigint not null`
- `papel_id bigint not null`
- `created_at timestamptz not null`

Chaves:

- `primary key (usuario_id, papel_id)`

FKs:

- `usuario_id -> usuarios.id`
- `papel_id -> papeis.id`

## Regras de Seguranca no Banco

### Minimo recomendado

- usuario da aplicacao com permissao apenas no banco da app
- backup automatizado
- restauracao testada
- nada de senha em texto puro

### Dados sensiveis

- nao armazenar token de acompanhamento em texto puro
- evitar IP puro se a politica de anonimato nao exigir isso
- anexos devem ficar em storage, nao em blob no banco
