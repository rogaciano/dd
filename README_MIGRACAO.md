# Migracao SQL Server 2008 -> PostgreSQL

Este diretorio agora possui dois caminhos:

- `migrate.py`: quando existir de/para por regra de negocio
- `migrate_copy.py`: quando o PostgreSQL espelha o schema do SQL Server e a carga pode ser quase 1:1
- `convert_schema.py`: gera um `schema_pg.sql` inicial a partir do `schemas.sql`

## Arquivos

- `migrate.py`: motor de migracao com de/para manual por tabela
- `migrate_copy.py`: copia tabelas diretamente em ordem de dependencia
- `convert_schema.py`: converte o DDL do SQL Server para um rascunho de DDL em PostgreSQL
- `table_mappings.py`: de/para por tabela
- `.env.example`: conexoes de exemplo
- `requirements.txt`: dependencias Python

## Banco de destino atual

Com a estrutura atual do `docker-compose`, use:

- `disque_denuncia_legado` para o espelho/importacao do SQL Server
- `disque_denuncia_novo` para a aplicacao nova

Para os scripts deste diretorio, o destino esperado e:

- `PGHOST=localhost`
- `PGPORT=5678`
- `PGDATABASE=disque_denuncia_legado`
- `PGSCHEMA=public`

## Antes de rodar

1. Instale o driver `ODBC Driver 17 for SQL Server` no Windows.
2. Crie e ative um ambiente virtual:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

3. Copie `.env.example` para `.env` e ajuste se necessario.

Se voce ainda nao criou `.env`, os scripts tentam usar `.env.example` como fallback e avisam no log.

Se o Windows tiver apenas o driver legado, use `MSSQL_DRIVER=SQL Server`.

Se voce ja estava usando a configuracao antiga, atualize o `.env` para apontar para:

```env
PGHOST=localhost
PGPORT=5678
PGDATABASE=disque_denuncia_legado
PGUSER=postgres
PGPASSWORD=postgres
PGSCHEMA=public
```

## Fluxo recomendado para copia 1:1

1. Gere o schema inicial do PostgreSQL:

```powershell
python convert_schema.py
```

2. Revise o arquivo `schema_pg.sql`.

Observacoes:

- o conversor ja troca `char` por `varchar`
- `text/ntext` viram `text`
- `image` vira `bytea`
- `bit` vira `boolean`
- `datetime/smalldatetime` viram `timestamp`
- `IDENTITY` vira `GENERATED BY DEFAULT AS IDENTITY`

3. Aplique o `schema_pg.sql` no banco `disque_denuncia_legado` do PostgreSQL.

4. Liste a ordem de carga:

```powershell
python migrate_copy.py --list-tables
```

5. Teste uma tabela:

```powershell
python migrate_copy.py --table usuarios --dry-run
```

6. Rode a copia:

```powershell
python migrate_copy.py --validate-counts
```

Se precisar recarregar tudo no destino vazio:

```powershell
python migrate_copy.py --truncate --validate-counts
```

Se um lote falhar, o script tenta isolar as linhas individualmente, registra as rejeitadas em `REJECT_LOG_PATH` e continua a carga. Para texto com byte NUL (`0x00`), o script remove esse byte antes de inserir no PostgreSQL.

## Como adaptar o de/para

Em `table_mappings.py`, cada item de `TABLE_MAPPINGS` representa uma tabela logica da migracao.

Campos principais:

- `name`: nome amigavel usado na linha de comando
- `source_query`: consulta no SQL Server
- `target_table`: tabela de destino no PostgreSQL
- `target_columns`: colunas do destino, na ordem do insert
- `conflict_columns`: chaves para `ON CONFLICT`
- `update_columns`: colunas atualizadas no upsert
- `truncate_before_load`: limpa a tabela antes de migrar
- `transform`: funcao que recebe um `dict` com as colunas da origem e devolve os valores do insert

Regra pratica:

- Se os nomes da origem ja baterem com os nomes do destino, use alias no `SELECT`.
- Se precisar converter tipo, normalizar texto, trocar nome de coluna ou montar novos campos, ajuste o `transform`.

Exemplo:

```python
{
    "name": "denuncia",
    "source_query": """
        SELECT
            Codigo AS id,
            Descricao AS descricao,
            DataCadastro AS criado_em
        FROM dbo.Denuncia
    """,
    "target_table": "public.denuncia",
    "target_columns": ["id", "descricao", "criado_em"],
    "conflict_columns": ["id"],
    "transform": lambda row: (
        row["id"],
        clean_text(row["descricao"]),
        row["criado_em"],
    ),
}
```

## Execucao

Listar os mapeamentos:

```powershell
python migrate.py --list-tables
```

Testar uma tabela sem gravar no PostgreSQL:

```powershell
python migrate.py --table usuario --dry-run
```

Migrar uma tabela com comparacao de contagem:

```powershell
python migrate.py --table usuario --validate-counts
```

Migrar tudo que estiver em `TABLE_MAPPINGS`:

```powershell
python migrate.py --validate-counts
```

## Observacoes para SQL Server 2008

- Como a instancia e antiga, eu deixei `ODBC Driver 17 for SQL Server` como padrao.
- O script prefere `host:porta` quando `MSSQL_PORT` estiver preenchido.
- Se a porta nao estiver acessivel e voce depender da instancia, deixe `MSSQL_PORT=` vazio e mantenha `MSSQL_INSTANCE=SQLEXPRESS`.
- Para servidor legado, `Encrypt=no` e `TrustServerCertificate=yes` costumam evitar atrito inicial.

## Proximo passo recomendado

Se o destino realmente puder espelhar a origem, use primeiro `convert_schema.py` + `migrate_copy.py`. Reserve o `migrate.py` apenas para as tabelas que exigirem ajuste manual.
